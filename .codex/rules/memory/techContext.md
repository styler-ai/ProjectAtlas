# Tech Context

Purpose: Note technical constraints or tooling requirements for ProjectAtlas.

- Rust/Cargo is the required implementation toolchain for ProjectAtlas 3.
- No Python implementation, Python scripts, Python tests, Python packaging, or Python CI should remain in the final ProjectAtlas 3 tree.
- Uses TOON for atlas output.
- `projectatlas init` writes the Rust configuration template; the current Rust CLI does not expose a `--no-detect-languages` option.
- ProjectAtlas 3 target stack is Rust with a Cargo workspace and strict gates: `cargo fmt --check`, `cargo check --workspace --all-targets --all-features`, `cargo clippy --workspace --all-targets --all-features -- -D warnings`, `cargo test --workspace --all-features`, doc tests, and rustdoc warnings.
- Documentation policy: use idiomatic Rust documentation (`//!`, `///`, `# Errors`) and fail builds on undocumented public APIs through rustdoc/lint gates.
- Current Rust crates: core, db, fs, service, cli, and symbols. MCP currently lives in the CLI binary through `projectatlas mcp`, the MCP adapter code is isolated in `crates/projectatlas-cli/src/mcp.rs`, and shared CLI/MCP runtime orchestration is isolated in `crates/projectatlas-cli/src/runtime.rs`; a future split crate may be useful if the adapter grows beyond the binary boundary.
- Interface decision: keep CLI and MCP as adapters over shared ProjectAtlas services. CLI provides deterministic local/CI testing; MCP provides Codex/OpenCode/Claude Code integration.
- Planned storage is `.projectatlas/projectatlas.db` with SQLite tables for nodes, purposes, symbols, health findings, metadata, and usage events.
- Storage decision: SQLite is the default because it is embedded, fast, cross-platform, transactional, and queryable; keep a storage boundary so future alternatives remain possible.
- Output decision: TOON is the default agent-facing output/export format because it is compact and token-efficient. ProjectAtlas runtime should use the official `toon-format` Rust crate for native TOON rendering, and tests/dev gates may validate representative payloads with the current official `@toon-format/cli` npm package as the spec-current oracle. CLI defaults to TOON. MCP uses JSON-RPC as the required transport envelope, but `atlas_*` tool text content should be TOON by default. JSON remains available only as an explicit tests/scripts/integration format.
- Rust toolchain choices: official/canonical crates where available: `toon-format` for TOON, `rmcp` for MCP, `rusqlite` with bundled SQLite, `ignore` for ripgrep-style `.gitignore` walking, `globset` for repository glob filtering, `rayon` for bounded parallel parser work, `notify` for watcher support, `tree-sitter` plus grammar crates for symbol parsing, `blake3`, `serde`, `clap`, and `thiserror`.
- Dependency governance: `deny.toml` is part of the release gate. CI/release must run `cargo deny check`; allowed licenses are explicit, unknown registries/git sources are denied, yanked/advisory issues fail, and duplicate transitive crate versions are visible warnings unless promoted later.
- Do not add random AI summarizer or purpose-generator crates. Current crate search shows generic summarizer crates exist, but not an official/canonical multi-language code purpose engine. ProjectAtlas should use official/canonical parser/indexing crates to extract facts, then apply deterministic ProjectAtlas summary policy in local Rust code.
- Scanner performance direction: use `ignore::WalkBuilder` and its parallel walker to keep `.gitignore` semantics while improving large-repo throughput. Add another walker only if measured evidence shows a real gap and the crate is canonical enough for the project quality bar.
- Parser strategy: keep symbol graph domain types in `projectatlas-core`, put grammar-heavy tree-sitter extraction in `projectatlas-symbols`, support Rust/Cargo as first-class target projects, use the official `toml` crate for Cargo manifest semantics, specialize Python, JavaScript, TypeScript/TSX, Java, Kotlin, C#, Go, Objective-C, Zig, C/C++, and keep broad fallback indexing for every other file family.
- Summary strategy: one-line node summaries are quick orientation only. The deep file summary command/tool must return structured deterministic facts comparable to and better than Code Index MCP `get_file_summary`, enriched with ProjectAtlas purpose state and rendered as TOON for agents.
- Deep summary parity details now include explicit exports, file docstrings/module comments, symbol docstrings/comments, package/module metadata, categorized symbols, stable line ranges, imports/dependencies, local calls, cross-file called-by references, and explicit live-source vs indexed-metadata fallback status. Remaining improvement is higher-quality language-specific extraction and stronger parser-specific import/export/call fidelity.
- Summary performance rule: `summary`/`atlas_file_summary` must not load all symbols or relations for large files. Use bounded SQLite section queries, exact count queries, and batched exact caller lookups; avoid leading-wildcard relation scans on the hot path.
- File/search/slice performance rule: `files --file-pattern`/`atlas_files.file_pattern`, `search`/`atlas_search`, and symbol slices must run through shared service/DB policy, not duplicate adapter policy. Folder/file navigation uses bounded SQL ranking and exact `globset` filtering without loading the whole node table; search uses indexed project-root resolution, repository-relative path matching, literal/regex/fuzzy matching, row streaming, early stop, and scanned/truncated telemetry. Symbol slices should fail closed on ambiguous same-name symbols until parent/kind/line disambiguation is supplied.
- SQLite path-filter rule: repository-relative folder path filters that use SQL `LIKE` must escape literal `%`, `_`, and escape characters and include an explicit `ESCAPE` clause. Treat path filters as exact path-prefix matching, not wildcard search syntax.
- Search storage rule: scan/watch refresh persists UTF-8 source text in SQLite `file_texts`, clears stale text rows for deleted/binary/non-UTF8 paths, and serves `search`/`atlas_search` from that index. Literal search can prefilter in SQL; regex and fuzzy matching stream over persisted indexed text without collecting the full corpus first. Exact outline/slice commands may still read live indexed source after the agent selects a file.
- Text-index cap rule: `scan.text_index_max_bytes` defaults to 2,000,000 bytes per file. Oversized files remain indexed as nodes and may still have purpose/metadata, but their full UTF-8 content is skipped in SQLite text search to keep large-repository memory and DB growth bounded.
- Scan policy rule: CLI `scan`/`watch` and MCP `atlas_scan`/`atlas_watch_once` must resolve scan config, ignore/exclude policy, and text-index caps through one shared runtime plan so adapter behavior cannot drift.
- Search default rule: search is intentionally case-insensitive by default for agent discovery. Use `--case-sensitive` or MCP `case_sensitive` only for exact-case work.
- Scan configuration rule: CLI scan, MCP scan, watcher refresh, and legacy `.purpose` cleanup must load normalized ProjectAtlas config and honor `[scan].exclude_dir_names`; this prevents generated/vendor/build-heavy directories from polluting large-repo maps, summaries, health findings, and token telemetry.
- Symbol build control rule: deep index builds expose `max_workers` and `timeout_seconds`; parser reads/extraction can run through `rayon`, but DB writes remain sequential for deterministic SQLite state.
- Caller correctness rule: deterministic `called_by` must be conservative. Attach a caller only when the relation target is exact, same-file, globally unambiguous, or path/module-alias resolved; do not attach suffix-only matches for duplicate symbol names.
- Watcher strategy: wire `notify` for event-backed recursive watching with debounce and ignore/exclude filtering. Keep the existing portable polling loop as fallback/test mode, but do not present polling-only behavior as final watcher parity.
- `notify` implementation guidance: create a `RecommendedWatcher` with the crate's recommended configuration, watch the canonical absolute project root recursively, debounce bursts before refresh, filter excluded/reserved paths before refreshing, and degrade to the existing polling loop if watcher construction or registration fails.
- Dependency strategy: prefer official or canonical Rust crates for standards, protocols, storage, parsing, watching, serialization, and token tooling; local code should focus on ProjectAtlas-specific product logic, orchestration, query policy, and agent workflow. Architecture review must flag unnecessary reimplementation.
- Path strategy: store repo-relative paths with forward slashes for stable DB/TOON/MCP output, but use `Path`/`PathBuf` for all host filesystem operations and tests on Windows-compatible path handling. Repo-relative file-key validation and repo-key-to-native conversion belong in `projectatlas-core` so CLI and service adapters share the same traversal/drive-prefix rules.
- Security path strategy: agent-facing file read tools (`outline`, `slice`, symbol slice, and MCP equivalents) must accept only project-relative indexed file paths. Reject absolute paths, drive prefixes, root paths, and `..` traversal before any file read.
- MCP/CLI expose overview, folder search, file search, compressed outline, structured file summary, text search, symbol graph, exact line/symbol slice, watcher refresh/status, health-check, lint, settings, reset-index, and usage overview/token-savings commands.
- MCP runtime rule: `projectatlas mcp` must run over the official `rmcp` stdio transport with Tokio timers enabled. E2E coverage should spawn the real binary, feed newline-delimited JSON-RPC requests, close stdin, and assert TOON text appears in tool results.
- Required repository-intelligence coverage: all-language discovery, search with filters/pagination/context lines, source summaries, symbol graph, exact code slices, watcher status, settings/cache inspection, and project switching.
- Required parity command surface: `projectatlas parity report --profile repository-intelligence` and MCP `atlas_parity_report`. This gate checks project-root metadata, structure index, purpose coverage, SQLite text index, symbol graph, watcher mode, health surface, token telemetry, CLI command surface, and actual MCP tool routes by name.
- Required token command surface: `projectatlas token` in CLI and `atlas_token_report` in MCP; slash command aliases such as `/projectAtlas:token` are harness adapters, not the canonical API.
- Token telemetry rule: count avoided wrong-folder exploration, wrong-file opens, and unnecessary full-code reads as the baseline ProjectAtlas prevented. CLI telemetry must measure the actual emitted payload for the selected format; MCP telemetry measures TOON tool text inside the JSON-RPC envelope. Token overview aggregation must stay SQL-backed so dashboards do not load all historical usage rows.
- Read-only telemetry rule: set `PROJECTATLAS_NO_TELEMETRY=1` when a reviewer, CI smoke, or read-only agent workflow must run orientation commands without writing usage rows to `.projectatlas/projectatlas.db`.
- Token TUI rule: `projectatlas token` remains structured agent output and `atlas_token_report` remains MCP TOON text. `projectatlas token --view tui` is the explicit human-facing terminal dashboard and must be covered by tests and live command smoke.
- Cargo gates replace the legacy Python gate stack; CI, hooks, and docs should use Rust commands only.
- DB-backed lint rule: when `.projectatlas/projectatlas.db` exists, `projectatlas lint` must also fail on unresolved SQLite health findings for missing, suggested, stale, duplicate, or repeated-temporary-folder issues. This preserves the old fail-fast purpose discipline without source/folder metadata pollution.
- Atlas-map config/manual path rule: config lists and non-source/manual map entries must be repository-relative; reject absolute paths, drive prefixes, root paths, and `..` traversal before existence checks or lint probes.
- Atlas-map determinism rule: `projectatlas map --force` must be idempotent for unchanged file/folder records so CI can run `map --force` followed by a map diff check. Preserve the existing `generated_at` value when the generated file/folder hashes match.
- Runtime cleanup rule: `reset-index` and MCP `atlas_reset_index` must default to dry-run and remove only selected local runtime files such as the SQLite DB and sidecars, plus MCP config only when explicitly requested.
- Plugin packaging target: installing the ProjectAtlas plugin should not only add skills; it should install or invoke the native Rust runtime and register MCP/tooling needed for ProjectAtlas 3. Current plugin carries `.mcp.json`, runtime installer scripts, and project-local MCP config generation.
- Plugin installer behavior: runtime installers default to the current working directory as the target project root, canonicalize explicit roots, prefer a local ProjectAtlas source checkout when present, otherwise download the GitHub Release binary for the platform using the release tag derived from the plugin manifest version, then fall back to that same tag with `cargo install --git https://github.com/styler-ai/ProjectAtlas --package projectatlas-cli --locked`, verify read-only `projectatlas --format json runtime-info` reports ProjectAtlas major version 3 or newer with MCP/TOON capabilities, skip stale wrappers, and write `.projectatlas/projectatlas.mcp.json` through `projectatlas --format json --db .projectatlas/projectatlas.db --config <config> mcp-config` when either `.projectatlas/config.toml` or flat `projectatlas.toml` exists. The generated MCP config includes a project-root `cwd` hint, while the MCP server also derives the root from config/DB metadata for hosts that ignore `cwd`. Release validation can force archive-only installation with PowerShell `-ReleaseBinaryOnly` or `PROJECTATLAS_RELEASE_BINARY_ONLY=1`, which must fail instead of falling back to Cargo if the published archive cannot be installed.
- Release packaging target: the `02-Release` workflow produces GitHub Release archives for Linux, Windows, and macOS; after publishing, release installer-smoke jobs must run the plugin installers against those published archive URLs before the release is considered fully verified.
- Release workflow security rule: workflow-dispatch release versions must enter shell steps through the `RELEASE_VERSION` environment variable, be validated as `^v[0-9]+\.[0-9]+\.[0-9]+$` before use, and never be interpolated directly into Bash or PowerShell scripts.
- CI PR gate rule: milestone enforcement should query live pull request metadata through `gh api` with `pull-requests: read`, because rerun pull_request event payloads can remain stale after a milestone is assigned.
- CI ProjectAtlas bootstrap rule: clean GitHub runners do not have `.projectatlas/projectatlas.db`; run `projectatlas scan .` against the checkout before DB-backed lint/parity gates, then run map and map-diff checks.
- Clean-checkout metadata rule: `.projectatlas/projectatlas.toon` is the committed portable atlas seed. When the ignored SQLite DB is absent, `scan` must import approved folder/file purposes from committed map rows before computing health/lint/parity.
- Legacy seed-purpose rule: `projectatlas init --seed-purpose` remains a migration/bootstrap helper and must write lint-valid `Purpose:` summary lines, not comment-only placeholders.
- Current installer verification: PowerShell runtime installer smoke passes on Windows and writes absolute DB/config paths. This Windows host has only the WSL `bash.exe` shim and no installed Linux distribution, so POSIX script execution must be proven by the GitHub release installer-smoke jobs on Linux/macOS.
- Review strategy: each substantial ProjectAtlas 3 loop should include a crate-choice/architecture check that asks whether an official/canonical Rust crate should replace local implementation. Local code is preferred only for ProjectAtlas-specific policy such as purpose governance, funnel orchestration, and health recommendations.
- TOON strategy: prefer the official `toon-format` Rust crate for serialization where its public API fits. If a narrow manual renderer remains for compatibility or migration output, keep it isolated, documented, and tested against the official TOON syntax expectations so ProjectAtlas stays spec-aligned. Generated map rows must quote/escape cells when paths or summaries contain commas, quotes, or control characters.
