# Active Context

Purpose: Capture current focus areas and any in-flight tasks for ProjectAtlas.

- Current focus is ProjectAtlas 3 Rust-only implementation and architecture hardening.
- Current implementation loop has replaced polling-only watcher parity with event-backed `notify` behavior, root-aware indexed file I/O, cross-platform folder and glob normalization, incremental watcher refresh for ordinary file changes, and shared service/DB paths for structured file-summary, direct glob file discovery, SQL-bounded ranked navigation, streamed search, health findings, and exact slice assembly; next work should continue remaining language-specific deep-index quality, measured very-large-repo performance evidence, and final publication readiness.
- Goal: reimplement ProjectAtlas and repository intelligence natively in Rust as one seamless ProjectAtlas 3 workflow for large codebases.
- First durable design artifact is `docs/projectatlas-3-architecture.md`.
- The Rust implementation must include a progressive context funnel, SQLite-backed purpose/index store, health-check/lint policy, token-savings telemetry, and a path toward complete repository-intelligence parity.
- Repository intelligence parity is a hard requirement: workspace selection, scan/refresh, all-language file discovery, search, summaries, symbols, code slices, watcher status, settings/cache inspection, and file content access must be covered with ProjectAtlas-native names and behavior.
- Code Index MCP source inspection showed its useful baseline tools are project selection, advanced search, glob file discovery, structured file summary, symbol body slicing, shallow/deep rebuilds, settings/status, and watcher configuration. Its `get_file_summary` is a structured deterministic view built from SQLite file rows plus symbol rows: language, line count, imports/exports/docstring, functions/classes/methods, signatures, line ranges, docstrings, and called-by metadata. ProjectAtlas must treat that as the minimum deep-summary baseline and exceed it with atlas purpose state, TOON, health, and token telemetry.
- The agent workflow must be atlas-first: project overview, relevant folders, relevant files, compressed outlines/summaries, symbols/slices, then exact code only when needed.
- Current Rust CLI includes repository-intelligence surfaces: `scan`, `overview`, `folders`, `files` with optional `--file-pattern` glob discovery, `outline`, `summary`, `search`, `slice`, `symbols`, `watch`, `settings`, `watch-status`, `reset-index`, `health-check`, `lint`, `token`, and `strip-legacy-purpose`.
- Current MCP surface is native Rust via `projectatlas mcp` and official `rmcp`, with `atlas_scan`, `atlas_overview`, `atlas_folders`, `atlas_files`, `atlas_outline`, `atlas_file_summary`, `atlas_search`, `atlas_slice`, `atlas_symbols_build`, `atlas_symbols`, `atlas_symbol_relations`, `atlas_health`, `atlas_token_report`, `atlas_settings`, `atlas_watch_status`, `atlas_watch_once`, `atlas_reset_index`, `atlas_strip_legacy_purpose`, and `atlas_purpose_set`.
- MCP verification must include a real `projectatlas mcp` stdio subprocess, not only in-process `rmcp` tests. The CLI runtime must enable Tokio timers because `rmcp` uses timer-backed service paths after initialization.
- Public naming must stay ProjectAtlas-native: CLI nouns like `folders`, `files`, `outline`, `slice`, `token`; MCP tools use `atlas_*` names such as `atlas_token_report`.
- Token-savings overview must be exposed through native CLI/MCP first (`projectatlas token`, `atlas_token_report`), with harness-specific aliases such as `/projectAtlas:token` allowed for Codex/OpenCode/Claude Code.
- Token-savings telemetry measures the context ProjectAtlas prevented the agent from wasting: wrong-folder exploration, wrong-file opens, and unnecessary full-code reads avoided by the overview -> folders -> files -> summary/outline -> exact-slice funnel. `projectatlas token` and `atlas_token_report` stay structured agent surfaces; `projectatlas token --view tui` is the explicit human terminal dashboard.
- TOON is the default agent-facing payload format. MCP must use JSON-RPC for transport, but `atlas_*` tool text responses should be compact TOON by default; JSON is an explicit machine-integration option, not the default agent path.
- Purpose and summary are separate. Purpose is the intended role of a folder/file and must be correct; summary is observed indexed content and may be deterministic. ProjectAtlas is for the agent harness, so missing purposes should be treated as lint/health failures, filled by the agent after inspection, and stored as agent-approved rather than left for a human approval bottleneck. Generated purpose guesses must not be silently treated as correct.
- File summaries are deterministic observed facts and may be generated from tree-sitter symbols/imports/dependencies/calls. Generated file-purpose one-liners may be useful suggestions, but they must remain `suggested` and trigger health review until the agent approves or corrects them. Folder purposes should remain agent-approved structural intent because folder responsibility is not reliably derivable from syntax alone.
- Every ProjectAtlas installation/session must establish the intended project root before scanning. The durable index belongs under that project root by default (`.projectatlas/projectatlas.db`) so different projects never share one global atlas.
- Deterministic health findings such as duplicate-purpose or repeated temporary folders should either lead to a real structure/purpose fix or be marked resolved in SQLite with agent rationale so the same accepted conflict does not repeat every run.
- Legacy Python ProjectAtlas implementation/tooling has been removed; do not reintroduce Python scripts, Python tests, Python packaging, or Python CI.
- Preserve the Codex plugin package in `plugins/projectatlas/` and keep it aligned with the ProjectAtlas 3 direction.
- ProjectAtlas plugin installation should bring the full ProjectAtlas 3 runtime/workflow: native binary or install path, MCP server config, TOON output, SQLite index support, health/lint, token telemetry, and clear funnel instructions. Current plugin installers default to the target project root, prefer a local ProjectAtlas checkout, then use pinned release binaries, and finally fall back to pinned Cargo/Git installation before writing `.projectatlas/projectatlas.mcp.json`.
- Before calling ProjectAtlas 3 complete, run a dedicated architecture-review subagent over the final diff for DRY/KISS/SOLID, crate boundaries, parser/runtime structure, performance, and agent-first workflow fit.
- Subagent review must also check official/canonical Rust crate usage. Local implementations are acceptable only for ProjectAtlas-specific policy, indexing decisions, purpose governance, health recommendations, and agent workflow orchestration.
- Cross-platform behavior is mandatory: repository paths stored in SQLite/TOON/MCP must be normalized with `/`, while filesystem access must use `Path`/`PathBuf` so Windows, Linux, and macOS all work.
- 2026-06-27 subagent source review of Code Index MCP found the remaining high-value parity gaps: ProjectAtlas needed exports/docstrings/package fields where available, deeper per-language extraction strategies, stronger cross-file call resolution, and event/debounce watcher semantics beyond portable polling. The first metadata gap is now partially closed with explicit `package`, `docstring`, `exports`, per-symbol `exported`, and per-symbol `documentation` fields in CLI/MCP summaries and symbol TOON.
- The current ProjectAtlas summary surface is intentionally richer than Code Index (`purpose`, `purpose_status`, `observed_summary`, `types`, `dependencies`, `calls`, `exports`, `docstring`, cross-file `called_by`), but the implementation must avoid being worse on Code Index's useful basics. Remaining focus is language-specialized extraction quality and parser-specific fidelity.
- Watcher status now advertises `notify`; continuous `projectatlas watch` should use the canonical Rust `notify` crate for event-backed recursive watching with debouncing/excludes and portable polling fallback, while MCP `atlas_watch_once` remains the safe bounded refresh surface.
- `watch-status` should probe whether `notify` can be constructed before reporting the active mode. It must not claim an active background watcher just because the SQLite DB exists. `watch` should run a full refresh for directory/root/ignore-rule events, but ordinary file create/modify/delete batches should use partial DB updates and symbol rebuilds for affected indexed files only.
- TOON conformance is a hard compatibility requirement. MCP cannot change its JSON-RPC envelope, but all agent-facing `atlas_*` tool text content should be official TOON-compatible output. When ProjectAtlas renders TOON locally, representative output should stay covered by tests or tooling against the official TOON implementation/spec.
- Generated compatibility maps must escape TOON tabular cells for legal paths/summaries containing commas, quotes, or control characters. The map writer keeps the compact row shape but quotes cells with JSON-compatible escapes when needed; non-source input supports both legacy unquoted rows and quoted cells.
- Final review found and fixed a security boundary issue: `slice`, `symbols slice`, and `atlas_slice` now reject absolute paths and parent traversal, require a project-relative indexed file in SQLite, and read telemetry baselines from the validated indexed path. `outline`/`atlas_outline` were put behind the same indexed-file boundary.
- Final review performance fixes: `summary`/`atlas_file_summary` now use targeted `load_node_by_path` instead of loading all nodes, the filesystem walker skips excluded directory subtrees with `WalkState::Skip`, and legacy `.purpose` cleanup sorts once after recursion.
- Latest architecture-review blockers were addressed: SQLite metadata now stores the canonical project root so outline/summary/search/slice/symbol-slice work from any current directory, and the watcher now carries debounced changed paths into partial scan/DB/symbol refresh instead of always rescanning the full repository after file events.
- File-summary, glob file discovery, indexed search, and exact slice product logic now live in `projectatlas-service`, not the CLI adapter. The service builds deterministic summaries from SQLite facts, uses bounded per-section queries plus exact SQLite counts instead of full file materialization, uses `globset` for repository-relative file/search filters with literal/regex/fuzzy matching and early-stop counters, streams SQLite text rows for regex/fuzzy search so pagination can stop early, rejects ambiguous duplicate symbol slices until parent/kind/line selectors are supplied, and exposes `source_status`/`source_error` when live source falls back to indexed metadata.
- Cross-file `called_by` is intentionally conservative: it uses exact target strings, path/module aliases, and global name counts so ambiguous same-name helpers are not attached to the wrong file. Empty `called_by` is preferred over a false deterministic caller.
- Legacy `.purpose` files have been stripped from the ProjectAtlas repo after import; current self-scan still reports 0 missing purposes. Do not recreate `.purpose` files except in migration tests/fixtures.
- Runtime/plugin wiring now includes `projectatlas mcp-config`, which emits a `.mcp.json`-compatible document using the current binary path, selected DB path, and config path when available. Plugin installers generate `.projectatlas/projectatlas.mcp.json` after installing or locating the native binary.
- Latest subagent parity/architecture audits found and the current implementation fixed three runtime gaps plus two large-repo/correctness hot-path issues: configured `[scan].exclude_dir_names` are now used by scan, MCP scan, watcher refresh, and legacy purpose cleanup; indexed search now supports native fuzzy matching; deep symbol builds now expose worker/timeout controls backed by `rayon` while SQLite persistence stays sequential; skipped live symbol candidates now clear prior symbols/relations/node summaries so old source facts cannot survive timeouts, max-byte skips, or non-UTF8 skips; and `overview()` now uses SQL aggregates instead of materializing every node. Remaining architecture-hardening follow-ups are a dedicated MCP adapter crate/application layer split, SQL/FTS-backed search optimization for very large repos, a shared repo-path abstraction to reduce duplicate validation code, and node-id-backed symbol graph storage for rename resilience.
- Latest token-architecture review found and the current loop is fixing telemetry correctness: CLI usage events must measure the actual emitted payload for TOON vs JSON, MCP usage events measure TOON tool text inside the JSON-RPC envelope, and `token_overview()` must aggregate in SQLite rather than materializing all usage rows for long-lived large repositories.
- Latest correctness-review blockers were addressed: scan excludes are now evaluated against repository-relative paths so checkouts under ancestor folders named `target`/`build` are not skipped; symlinked files are skipped by default instead of followed outside the repository; approved file purposes become `stale` when content hashes change; health/lint surface stale/missing/suggested/duplicate-purpose issues from SQLite; `watch --once` preserves rich unchanged summaries; and watcher refresh detects newly added folders/files, refreshes SQLite text, and builds symbols for new files.

- Search is intentionally case-insensitive by default for agent discovery; use `--case-sensitive`/`case_sensitive` only when exact casing matters.
