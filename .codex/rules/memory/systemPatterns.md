# System Patterns

Purpose: Record key architectural patterns in ProjectAtlas.

- Legacy atlas behavior reads Purpose headers, `.purpose` files, and `projectatlas-nonsource-files.toon`; Rust must preserve the useful migration behavior without keeping Python as the implementation.
- `projectatlas init` now auto-detects language extensions to seed scan + purpose style config.
- ProjectAtlas 3 target pattern replaces source/folder metadata files with SQLite-backed nodes, purposes, symbols, health findings, and usage events.
- ProjectAtlas 3 context access must be funnel-shaped: startup context -> relevant folders -> relevant files -> compressed outline -> exact code slice.
- The preferred funnel is project root -> overview -> folder purpose -> file purpose plus observed one-line summary -> structured file summary -> symbol graph -> exact slice. Agents should not jump to full-file reads before the atlas and structured summary layers.
- ProjectAtlas 3 should provide native CLI and MCP surfaces; MCP tools should bias agents away from full-file reads until folder/file/outline steps are exhausted.
- ProjectAtlas 3 architecture should be core-first: scanning, indexing, querying, health, lint, and telemetry live in reusable core crates; CLI and MCP are adapters over the same services.
- Code intelligence and purpose intelligence must be one ProjectAtlas pipeline, not two adjacent tools. The same scan/index state should drive folder purpose, file purpose, language metadata, source summaries, symbols, search, slices, health, lint, and token telemetry.
- Structured file summary is a first-class deep-index output. It should be deterministic and source-derived: file metadata, language, line count, purpose status, observed summary, imports/dependencies, functions, methods, classes/types, signatures, line ranges, and calls/called-by. Purpose is not part of deterministic summary; it is durable agent-approved intent.
- Structured file summary should be built from persisted SQLite index state, not from ad hoc live parsing in every request. Scan/index refresh owns extraction; summary and MCP tools query indexed facts so large repositories stay fast and predictable.
- Import-derived `called_by` disambiguation must run after ambiguous direct-name checks. If a call target equals a symbol name but the name is globally ambiguous, do not return false immediately; allow persisted import relations to prove a unique caller first. Module aliases must treat package entries and compound extensions consistently, including `index.*`, `__init__.py`, `mod.rs`, and `.d.ts`.
- Glob file discovery, indexed search, and exact source slices should also be shared service-layer behavior, not adapter-local policy. File discovery and search use repository-relative `globset` filters; search adds literal/regex/fuzzy line matching, bounded pagination, early stop, and scanned/truncated counters. Symbol slices reject ambiguous duplicate names until the agent supplies parent, kind, or line selectors.
- Broad text search should use persisted SQLite file text, not repeated O(N) live disk reads. Scan/watch refresh owns `file_texts` updates, literal search may use SQL prefiltering, regex/fuzzy remain service-level line matching over indexed text, and exact outline/slice remain explicit live-source escalation for selected indexed files.
- Runtime scan paths must honor normalized ProjectAtlas config, especially `[scan].exclude_dir_names` and `[scan].exclude_path_prefixes`, across CLI scan, MCP scan, map/lint, watcher refresh, indexed reads/search, and legacy cleanup. Default scanner options are only acceptable in scanner crate tests or explicit no-config contexts.
- Watcher event paths must be revalidated as safe repository-relative keys before incremental refresh. Parent traversal, empty, current-directory, root-prefix, or outside-root event paths are ignored even if a lexical path strip could produce a string.
- Legacy purpose-map import must be forgiving at the migration boundary. It should import only paths that are present in the fresh scan, count stale/deleted/excluded map rows as skipped stale imports, and avoid exposing low-level SQLite no-row errors for automatic migration. Direct agent/user purpose writes should still fail when the target path is not indexed.
- Deep symbol parsing may run in parallel via `rayon`, but SQLite persistence should remain deterministic and sequential unless the DB layer is explicitly redesigned for concurrent writes.
- Language intelligence should use a strategy boundary: generic tree-sitter traversal provides broad coverage, while language-specific extractors add better imports, exports, docstrings/comments, package/module data, signatures, and call resolution for Rust, Python, JavaScript/TypeScript, Java, Kotlin, C#, Go, Objective-C, Zig, C/C++, and fallback file families.
- The v0.3.2 symbol architecture should make that strategy boundary concrete inside `projectatlas-symbols`: keep the public API stable, first move the language augmentation cluster into focused internal modules, and keep the generic tree-sitter parser spine stable until that lower-risk split is green. New language fidelity should land behind those module boundaries instead of expanding one monolithic source file.
- Large-codebase performance is a first-class pattern: `.gitignore`-aware parallel walking, content hashes, incremental refresh, pagination, compact TOON responses, SQLite indexes, and exact-slice escalation should be default behavior.
- Health-check is a first-class subsystem, not a report bolted onto lint. It should produce severity, category, path, related path, message, and recommendation.
- Lint becomes policy over the index: fail on missing/stale purposes, stale index state, and configured high-severity health findings without modifying source files.
- Health should treat generated purposes as review-required suggestions, not approved facts. Duplicate-purpose checks should use approved purposes only so generated guesses do not create false cleanup work.
- Approved file purposes become stale when content hashes change; agents must inspect the updated summary/context and approve or correct the purpose in SQLite. Health and lint should surface stale purposes until resolved.
- Project root selection is part of setup. Each project has its own `.projectatlas` state and SQLite index; plugin instructions must make agents establish the root before scan/index calls.
- Exact source access is an indexed-file privilege, not arbitrary filesystem access. CLI and MCP outline/slice tools must reject absolute paths and parent traversal, then verify the project-relative file exists as an indexed `NodeKind::File` before reading.
- Repository-intelligence behavior should be implemented as Rust-native ProjectAtlas functionality, tracked by behavior coverage rather than copied implementation or copied naming.
- Native display/metadata path normalization should be centralized in `projectatlas-core`. CLI, DB, MCP diagnostics, and settings should share one helper for slash normalization and Windows extended path prefix cleanup instead of duplicating `\\?\`/UNC handling.
- A brief README acknowledgement can say ProjectAtlas 3 was inspired by code-index MCP repository indexing ideas, but code, command names, class/struct names, crate names, and MCP tools stay ProjectAtlas-native.
- Watcher behavior should prefer event-backed `notify` with debouncing and excludes, then fall back to polling when platform support is unavailable. MCP and CLI should expose the same refresh/status model.
- Watcher refresh should preserve the atlas funnel: file events trigger a fast index refresh, persisted summaries/symbols/relations update in SQLite, and subsequent CLI/MCP reads query the updated store instead of reparsing files ad hoc.
- Watcher refresh should auto-detect new files and folders. Full refresh handles directory/root/ignore-rule events; partial refresh upserts changed nodes, clears absent paths, refreshes SQLite text, and rebuilds symbols for affected files.
- MCP transport stays standards-compliant JSON-RPC. TOON belongs inside the returned text content for agent-facing tools, so Codex/OpenCode/Claude Code read compact structured payloads while MCP clients still receive valid protocol messages.
- Native MCP is an adapter module over the same runtime/service/database behavior, not a separate product path. Keep `projectatlas mcp` route definitions, MCP schemas, and stdio startup isolated in `mcp.rs`; keep scan/watch/text-index/symbol/reset/settings/purpose orchestration in `runtime.rs`; keep `main.rs` as the executable CLI shell.
- Runtime identity and root binding are part of the adapter contract. `runtime-info` must remain read-only and side-effect-free. Generated MCP configs should include absolute runtime/DB/config paths plus a `cwd` hint, discover `.projectatlas/config.toml` or flat `projectatlas.toml` from the selected DB/project root, and path-less MCP tools must resolve the intended root from config, indexed DB metadata, or `.projectatlas/projectatlas.db` before process cwd.
- ProjectAtlas MCP is project-local. Root-changing MCP arguments such as `atlas_scan.path`, `atlas_watch_once.path`, and legacy cleanup paths must resolve to the server-bound project root; use a separate MCP server/DB/config for another repository.
- Config-root resolution must treat bare `.projectatlas/config.toml` and explicit `./.projectatlas/config.toml` as equivalent. When that config declares `[project] root = "."`, an empty grandparent path means the current working directory/project root, not an empty filesystem path.
- MCP stdio transport follows the official MCP newline-delimited JSON-RPC framing. Do not document or test ProjectAtlas MCP as if it used LSP-style `Content-Length` framing on stdio.
- Every substantial ProjectAtlas 3 implementation loop should end with two checks: strict Rust gates and a subagent review covering architecture, official crate choices, path portability, performance, and agent-first workflow ergonomics.
- Architecture follow-ups from the 2026-06-27 review: split the MCP adapter or create a cleaner shared application layer as `main.rs` grows, push more large-repo counts/search into SQL or SQLite FTS5 before considering a separate engine, centralize repo-path validation in core, and consider node-id-backed symbol graph storage for rename resilience.
- Plugin runtime setup is part of the product surface. Installers should establish the project root from the caller's current working directory, install or locate the native `projectatlas` binary, emit an MCP config with absolute `--db` and `--config` paths plus a project-root `cwd` hint, and leave agents with a clear atlas-first command sequence.
- Plugin runtime verification should use `projectatlas --format json runtime-info` as the stable contract: project `ProjectAtlas`, major version 3 or newer, capability `mcp`, and text format `TOON`. Avoid brittle help-text matching.
- Parity reporting is a first-class release gate. `projectatlas parity report` and `atlas_parity_report` should stay machine-checkable and fail when the live index lacks required structure, purpose, text-index, symbol, watcher, health, telemetry, CLI, or MCP readiness.
- CI/release freshness should lint the checked-in atlas state before regenerating maps. Regeneration is allowed as an explicit maintenance step, but stale map output should not be hidden by automatic generation before the lint gate has a chance to fail.
